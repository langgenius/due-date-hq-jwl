# Deadline status — meaning, journey, interaction (2026-05-23)

This doc thinks through the 6 canonical deadline statuses from the CPA's
perspective: **what each status means** in their head, **how they mark
it** (entry & exit triggers), **what they need to do** while in it, and
**how the panel should support that journey**.

Sets the design contract for the obligation panel's stage card. The
mechanical task wiring landed earlier today; this doc is the design
brief behind it (and the punch list of what's still missing).

## The 6 statuses, in journey order

```
NOT STARTED → IN REVIEW → FILED → COMPLETED
                ↕             ↕
              WAITING       (rejected sub-state → unwinds back to IN REVIEW)
              BLOCKED
```

`waiting_on_client` and `blocked` are interruption states that branch
off IN REVIEW and return to it once the interruption clears. `filed
(rejected)` re-routes back to IN REVIEW via the `markFiledRejected`
mutation. Otherwise the flow is one-direction.

---

## 1. Not started (`pending`)

### What it means

> "This deadline exists, but nobody has touched it yet."

The row was generated by the rule library (or imported / created
manually) and is sitting in the queue waiting for someone to start
work. **It's a queue state, not a work state.** No prep effort has
happened yet.

### Entry triggers

- Rule library auto-generates the deadline when its activation rule
  fires.
- CSV / manual create.
- Year roll-over from completed prior-year row.

### Exit triggers (CPA action)

- **Primary**: "Start preparation" → moves to IN REVIEW.
- **Backwards**: "Not applicable" (mark `not_applicable`) — only if
  the rule turned out to be wrong for this client.

### What the CPA needs to do

- Confirm the engagement exists (e.g. signed engagement letter on
  file for this client).
- Assign a preparer to the row.
- Glance at the form/jurisdiction/due-date to make sure it makes
  sense for this client (rule library generated it — does it look
  right?).

### What the panel should show

- **The deadline facts**: form name, tax year, jurisdiction, due
  date, days-until-due (tone-coded if overdue or near-due).
- **Pre-work reminders**: engagement letter, assignment.
- **One primary action**: Start preparation.

### Current state ✅ mostly there

- Pre-work reminders show as small text. They aren't actionable
  (clicking "Assign preparer" doesn't open a member picker — we don't
  surface assignee selection from the drawer yet). Gap, but
  acceptable for now.
- The deadline facts live in the drawer's header + Dates panel.
  Stage card itself doesn't duplicate them — good (information
  density).

## 2. Waiting (`waiting_on_client`)

### What it means

> "I started, but I can't continue until the client gives me X."

The blocker is **THIS client's missing input** — typically W-2s,
1099s, K-1s, or books cleanup. CPA can resume the moment the input
arrives. **It's a client-input-pending state**, not a project blocker.

Sub-status (`prepStage`) refines:

- `waiting_on_client` — generic outstanding docs
- `waiting_on_third_party` — K-1 from another firm, brokerage 1099,
  etc.
- `bookkeeping_cleanup` — books need to be fixed before prep can
  continue

### Entry triggers (CPA action)

- From IN REVIEW: CPA spots missing input, sends a readiness request
  to the client, changes status to Waiting.
- From NOT STARTED (rare): CPA pre-emptively requests docs before
  even starting prep.

### Exit triggers

- **Primary**: "Mark docs received" → back to IN REVIEW.
- **Backwards**: rarely — possibly back to NOT STARTED if scope
  changes.

### What the CPA needs to know

- **WHAT** is outstanding (which checklist items).
- **WHO** is the client contact (email/phone reachable from the
  contact-meta row higher up the page).
- **WHEN** was the last reminder sent (and how many have we sent).
- **Whether the client responded** to the readiness request at all.

### What the panel should show

- Outstanding documents list (or count + "view all").
- Sub-status label so CPA knows whether it's the client themselves,
  a third party, or bookkeeping.
- Routing to the Readiness tab (where the full checklist lives and
  reminders can be sent).
- One primary action: Mark docs received.

### Current state — partially there

- ✅ Sub-status label rendering (Documents from client / Third-party
  docs / Bookkeeping cleanup / Ready for prep).
- ✅ "Send readiness request" routes to Readiness tab.
- ✅ Primary "Mark docs received" wired.
- ❌ **Gap**: outstanding documents count NOT inline on the stage
  card. CPA has to switch to the Readiness tab to see what they're
  waiting on. Should surface a count + the first 2-3 outstanding
  item labels inline.
- ❌ **Gap**: no "last reminder sent X days ago" indicator. Punted
  to the Readiness tab.

## 3. Blocked (`blocked`)

### What it means

> "An EXTERNAL factor — usually another return — has to land before
> I can move this one."

Different from Waiting. Waiting = client input. Blocked = systemic
dependency. Classic example: a partner's personal 1040 is Blocked by
the partnership's 1065 (because the partnership return generates the
K-1 the partner needs). The blocker can be another obligation OR an
external event (authority opens filing window, third-party software
rolls out a form).

### Entry triggers

- CPA realizes there's an upstream dependency, marks Blocked, sets
  `blockedByObligationInstanceId` to point at the upstream row.

### Exit triggers

- **Primary**: "Mark unblocked" → back to IN REVIEW.
- Could auto-clear when the blocker row reaches COMPLETED (future
  enhancement — currently CPA has to manually unblock).

### What the CPA needs to know

- **WHAT** is blocking (which obligation: form + client + due +
  current status).
- **WHEN** the blocker is expected to clear (the blocker's own
  due/status tells you).
- **WHO** owns the blocker (if a teammate, the CPA can ping them).
- A direct way to OPEN the blocker without searching for it.

### What the panel should show

- A "Blocked by" card: blocker form, blocker client (often the same
  client — partner ↔ partnership), blocker due date, blocker current
  status. Clickable → opens the blocker's drawer.
- One primary action: Mark unblocked.

### Current state — partially there

- ✅ Primary "Mark unblocked" wired.
- ✅ "Open blocking obligation" now wired to open the blocker drawer
  (landed earlier today).
- ✅ The drawer header has a `BlockedByChip` showing the blocker's
  form code; the stage card surfaces its sub-status as "Upstream
  obligation."
- ❌ **Gap**: the stage card itself doesn't show the blocker's full
  context (form + client + due + status). CPA gets a one-word "Upstream
  obligation" sub-status and a routing link, but nothing about WHICH
  upstream obligation is blocking. They have to click through to find
  out.

## 4. In review (`review` / `in_progress` / `extended`)

### What it means

> "Active work. I'm preparing this return."

This is the biggest stage and has the most internal sub-states.
`review` covers everything from "ready_for_prep" through "approved
and ready to file." Sub-status (`prepStage` + `reviewStage`) carries
the real granularity:

- `prepStage`: ready_for_prep → in_prep → prepared
- `reviewStage`: ready_for_review → in_review → notes_open → approved

`extended` is a special variant: an extension was filed, so the
internal due date moved out 6 months but the row is still in active
prep on the new timeline. `in_progress` is legacy from before
lifecycle v2; should rarely appear.

### Entry triggers

- From NOT STARTED: "Start preparation"
- From WAITING: "Mark docs received"
- From BLOCKED: "Mark unblocked"
- From FILED (rejected): `markFiledRejected` unwinds back here

### Exit triggers

- **Primary**: "Mark filed" → FILED
- Sideways: to WAITING (if a new doc gap is found) or BLOCKED (if an
  upstream dependency appears).
- Backwards: "Mark not_applicable" if the rule turns out to be wrong.

### What the CPA needs to do

- Prep the return (calculate, fill the form, cross-check).
- Send for reviewer sign-off (if firm has reviewer role).
- Resolve any review notes.
- Get 8879 signature from client.
- File.

### What the panel should show

- **Sub-status visualization**: where in the prep ↔ review pipeline
  the row is. Mirror the e-file pipeline strip we already use on
  FILED.
- Routing to Evidence tab (where 8879 lives).
- Primary "Mark filed".

### Current state — partially there

- ✅ Compact In Review step strip (Preparing return / Reviewing return /
  Ready to file), derived from prep/review sub-status fields.
- ✅ Primary "Mark filed" wired.
- ✅ "Get 8879 signed by client" routes to Evidence.
- ✅ Done-this-stage audit chronology (shown in the screenshot).
- ✅ Notes-open annotation when reviewer feedback is active.

## 5. Filed (`done`, sometimes `paid`)

### What it means

> "I submitted to the authority. Now I'm waiting for the next thing
> to happen."

**Filed ≠ Done.** That's the central insight from the canonical
product spec. The return has been submitted but the work isn't over.
Sub-states walk the e-file pipeline:

```
authorization_requested → authorization_signed → ready_to_submit
  → submitted → accepted (or rejected) → final_package_delivered
```

`paid` is the parallel branch for payment-style obligations
(estimates, sales tax remittance):

```
estimate_needed → client_approval_needed → scheduled → confirmed
```

Both end at COMPLETED.

### Entry triggers

- From IN REVIEW: "Mark filed"

### Exit triggers

- **Primary**: "Confirm authority acceptance" → COMPLETED (or
  "Mark obligation complete" from terminal sub-states)
- **Backwards**: "Record authority rejection" → unwinds to IN REVIEW
  via `markFiledRejected`

### What the CPA needs to know

- Which sub-state the row is in.
- If `submitted`: when did we submit, when do we typically hear
  back?
- If `rejected`: why (rejection reason from the authority).
- If `accepted`: have we delivered the final package to the client?

### What the panel should show

- 6-step pipeline visualization (we have this).
- Sub-state-specific actions:
  - `submitted` → Confirm acceptance (primary) + Record rejection
  - `accepted` / `paper_filed` → Mark obligation complete + manual
    reminder to deliver the final package
  - `rejected` → Unwind to In review (primary)
  - everything before submission → manual reminders only (no backend
    sub-status RPC yet)

### Current state ✅ mostly there

- ✅ Pipeline visualization for both e-file and payment branches.
- ✅ Wired primaries on every sub-state that can advance to terminal.
- ✅ Manual reminders for offline-only steps.
- ❌ **Gap**: no rejection-reason field surfaced for `rejected`
  sub-state. The CPA sees the chip but not why the IRS rejected.
- ❌ **Gap (backend)**: no sub-status RPC, so 8879 → submitted is
  offline-only. Real CPAs would expect to flip those.

## 6. Completed (`completed` / `not_applicable`)

### What it means

> "Done. Filed, accepted, package delivered. Or it didn't apply this
> year."

Terminal. No further action required. The row sticks around as
historical record and as a reference for next year's planning.

### Entry triggers

- From FILED: "Confirm authority acceptance" / "Mark obligation
  complete"
- `not_applicable` is set directly from any non-terminal status when
  the rule turns out not to apply.

### Exit triggers

- None normally. Re-opening would be a deliberate audit action
  (e.g., authority sends a deficiency letter months later) — handled
  via the status picker as a manual override.

### What the CPA needs to see

- Final outcome summary: when filed, when accepted, total turnaround.
- Where to find the workpapers (archive link).
- Audit trail — a record of how this got here.

### Current state — minimal

- ✅ Stage card shows "Archive workpapers" as a manual reminder.
- ❌ **Gap**: no summary of the key dates (filed date, accepted
  date, total cycle time). CPA needs to inspect the Dates panel
  separately. For a TERMINAL stage, the summary should be inline.
- ❌ **Gap**: no link to the archived workpapers / final package.

## User journeys

### Happy path (the bulk of work)

```
NOT STARTED
  → Start preparation
IN REVIEW (prep → review → 8879)
  → Mark filed
FILED · authorization_requested
  → manual: client signs 8879
FILED · submitted
  → Confirm authority acceptance
COMPLETED
```

### Waiting branch

```
IN REVIEW
  → (CPA spots missing doc) status picker: Waiting
WAITING · documents from client
  → Send readiness request (route to Readiness tab) → reminder sent
  → (client responds) Mark docs received
IN REVIEW (resumes)
```

### Blocked branch

```
IN REVIEW
  → (CPA spots upstream dependency) status picker: Blocked,
     link blockedByObligationInstanceId
BLOCKED · upstream obligation
  → Open blocking obligation (routes to blocker's drawer)
  → (blocker completes; CPA returns) Mark unblocked
IN REVIEW (resumes)
```

### Rejected sub-flow

```
FILED · submitted
  → Record authority rejection → markFiledRejected
IN REVIEW (efile_rejected_at set, Rejected chip rendered)
  → CPA fixes the return
  → Mark filed (again)
FILED · ... → Confirm acceptance → COMPLETED
```

## Gaps the panel should fill (priority-ordered)

### P0 — High user-confusion cost

1. **Blocked: inline blocker card.** Currently CPA sees "Upstream
   obligation" sub-status and a routing link. They don't know WHICH
   upstream obligation is blocking until they click. Surface form +
   client + due + status inline on the stage card.

2. **Waiting: outstanding docs count + first items inline.** Today
   CPA has to switch to the Readiness tab to know what they're
   waiting on. Show "3 outstanding: W-2 · 1099-DIV · K-1" inline.

### P1 — Workflow visibility

3. **In review: compact workflow visibility.** Show the CPA-facing path
   as Preparing return → Reviewing return → Ready to file, with
   notes-open shown as an annotation on Reviewing return. Keep the
   detailed prep/review flags in audit data rather than exposing them
   as separate steps.

4. **Completed: key-dates summary.** Show filed date / accepted date
   / total turnaround time on the terminal stage card. Saves the CPA
   from inspecting the Dates panel to answer "when did this close?"

### P2 — Backend-gated

5. **Filed: sub-status mutations.** Need new RPC procedures
   (`updateEfileState`, `updatePaymentState`) so the pre-submission
   sub-states can actually advance through the UI. Today they're
   manual reminders only.

6. **Filed (rejected): show rejection reason.** Backend already has
   `efile_rejected_at`; need a `efile_rejection_reason` field +
   surface on the stage card.

### P3 — Niceties

7. **Not started: "Assign preparer" → member picker.** Currently
   text-only reminder; could route to a member picker.

8. **Blocked: ETA / auto-unblock.** When the blocker row reaches
   COMPLETED, surface a banner "blocker resolved — mark this
   unblocked?" or auto-clear (with audit).

---

## Decision: what to ship in this batch

Implementing P0 (blocked card + waiting count) gives the biggest
clarity win for two stages that today require a tab switch to
understand. P1 (in review pipeline + completed summary) are bigger
design pieces — propose those next round.

P2 (backend sub-status RPCs) is its own server task.
P3 are followups.
