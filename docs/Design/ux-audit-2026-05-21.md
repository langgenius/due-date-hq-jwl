## Cross-cutting themes

### 1. Vocabulary fragmentation

The product cannot tell the same story twice the same way.

- **Pulse alerts** appear with three different names + icons + dismissal
  semantics across Dashboard banner, Inbox, and `/rules/pulse`.
  PulsingDot vs AlertTriangle vs hairline strip. "Pulse alert" vs
  "Need your decision" vs "Heartbeat strip." A new preparer cannot tell
  if these are the same thing.
- **Status taxonomy** is mid-migration. The route filter parses
  `pending | in_progress | waiting_on_client | review`; the actions list
  switches on `pending | in_progress | waiting_on_client | blocked | review
| completed`; the canonical 6-state target
  (`not_started/waiting_on_client/blocked/in_review/filed/completed`)
  matches neither.
- **Same color, different urgency.** RejectionChip and BlockedByChip both
  render in `text-text-destructive` on
  `bg-state-destructive-hover`. Rejection is "IRS sent it back"
  (actionable now). Blocked-by is "waiting on a dependency"
  (actionable but informational). Identical visual weight = noise.

### 2. Redundancy stack

Each surface has more chrome than content.

- **Dashboard:** v1 + v2 both ship in the route. `useDashboardV2`
  defaults to true on this branch but the v1 path (`DashboardTriagePanel`,
  `NeedsReviewBanner`, `ProjectedRiskInline`, full filter facets) still
  mounts conditionally. `?dashboard=v1` URL resolves a different product.
- **Obligations queue:** 8 columns at peer visual weight (client, state,
  tax type, due, projected risk, evidence, status, +chips). No primary
  anchor, no muted secondary, no progressive disclosure. The row is a
  spreadsheet, not a queue.
- **Rule library:** matrix view stacks 5 horizontal bars before the
  table: CoverageSummaryStrip + SourcesSummaryStrip + StatsStrip +
  SourceStatusBanner + EntityCoverageLegend. Active/pending counts appear
  twice on the same page (top summary strip + StatsStrip).

### 3. Invisible affordances

Power-user features exist but only power users know they exist.

- J/K/Esc hotkeys registered in obligations queue and rule review mode —
  zero in-UI hints. Priya enters review mode and has no way to discover
  she can press `j`.
- **Inline drawer close affordance:** modal mode gets Sheet's X + Esc +
  backdrop click. Inline mode has only a ghost "Close" button buried in
  the footer at `obligations.tsx:4430`. Three viewports, three different
  close contracts.
- **Rule library toggle:** `<Button variant="outline">View all rules →</Button>`
  floats at the strip's right edge. ChevronRight icon implies "navigate
  elsewhere," not "switch view." Users don't realize it's a persistent
  mode toggle until they click.

### 4. Missing safety net on destructive moves

- Status changes fire `updateStatusMutation` immediately. Success toast
  shows the audit hash but no `Undo` action. For destructive moves
  (`completed` → anything, `filed` → `in_review`) this is reversible in
  audit but invisible to the user.
- "Hide" on Dashboard System Status is session-only — no way to bring it
  back.
- No bulk-undo on bulk owner reassign.

## Provocative questions (don't answer yet)

1. **Is "Actions this week" actually a dashboard, or is it the Obligations
   queue with a 7-day filter and a sexier card?** If it's the same query,
   why does the user need both surfaces?

2. **Why does System Status sit above Pulse alerts on the Dashboard?**
   Source-health is an internal engineering concern; Pulse is client-facing.
   Reversing the order would put the loudest thing at the most-important
   thing.

3. **Eight columns + chip variations + status pill + due stack on the
   Obligations row: when does this stop being a queue and start being a
   spreadsheet?** Linear, Asana, Height each chose 4-5 columns and pushed
   the rest to row hover. What's the cost of saying no?

4. **If the two summary strips and the StatsStrip all show the same data,
   why do all three exist?** What does the Rule library lose if you delete
   two of them?

5. **The Dashboard has zero personalization by role.** A partner and a
   preparer see the same page. Should a preparer even see "Need your
   decision" (a review-tier action), or should that be partner-only?

## What's working (capture before changing it)

Per the critiques, these are real product insights that should NOT be
touched in the next pass:

- **Dashboard action-led rows** (verb-prompt over data) — tells juniors what to
  _do_, not what to _look at_. Rare and good.
- **Dashboard ExposureStrip tone discipline** — only `blocked` is destructive;
  everything else neutral. Dashboard refuses to be a fire alarm.
- **Obligations stacked Due cell** — relative pill + absolute date + statutory
  divergence in ~3 lines without a legend.
- **Obligations inline-vs-modal hybrid mechanics** — no focus trap in inline
  mode lets J/K swap panel content without close/open. Real product
  insight.
- **Rule library review-mode workspace card** — collapsing the page header,
  mounting a 320px queue rail + detail panel in one bordered surface is a
  sharp move. The feature's actual point of view.
- **`OriginBreadcrumb` pattern** with per-page Clear semantics — rare to see
  drill-ins respect return paths this cleanly.

## What this audit doesn't cover

- **Mobile.** None of the three critiques opened DevTools below md
  breakpoint. The breadcrumb at 11px on a 375px viewport, drawer panels
  that shift between inline/modal/full-screen — needs its own pass.
- **Empty / loading / error states.** Spot-checked, not exhaustive.
- **Tab title + sidebar active state alignment** when in compact mode
  (rule review). Four wayfinding signals, only one tells the truth.
- **Notification channel consistency beyond Pulse.** Reminders, audit
  events, system status, billing alerts all surface somewhere — the same
  fragmentation likely exists.
- **Onboarding / first-run.** Empty product is invisible in this audit.
