## 1 · Executive summary

The app has accumulated a strong visual vocabulary — tone-coded badges, hairline tables, careful empty states, monogram avatars, StateBadge motifs — and a clear "GitHub-density" north star (touched 17 times in code comments this commit cycle). What it has **not** done is converge on a single rhythm across surfaces. The dashboard h1 sits at 20 px (`text-xl`), every other route's `PageHeader` h1 sits at 24 px (`text-2xl`), the obligations queue title sits inside a `ConceptLabel` info wrapper that nothing else does, and the rules library wraps the whole page in a separate `RulesPageShell` primitive. Five top-level pages = five different ways to start. Inside surfaces the inconsistency continues: `/clients` stat-tiles are 12 px (`rounded-md`) + `text-xl` value, `/clients/[id]` summary strip tiles are 12 px (`rounded-xl`) + `text-xl` value but with 30 %-opacity inline-style labels, `/opportunities` tiles are 6 px (`rounded-md`) + `text-2xl` value with bordered-card chrome, `/rules/library` `StatTile` is 6 px + `text-xl`. **The same primitive concept ("compact metric tile") ships as four parallel implementations** — none of which is exported from a shared module. This is the most fixable theme in the entire audit.

Beyond the tile zoo, `/clients/[clientId]` is doing **too much work in one component** (the 4,532-LOC file is 60% of the entire clients feature footprint), and the page can be split into the table (where a CPA scans clients) and the detail (where they read about one). Today they live in the same file as `ClientFactsWorkspace` + `ClientDetailWorkspace` with deeply-shared helpers; the file's own comments call out a `TabSection` retrofit, but the underlying structural cost — three nested `<section>` wrappers (`<div xl:flex-row>` → `<div flex-1 flex-col gap-6>` → `<section flex flex-col gap-6>` → tabs) before any content paints — visibly shows in screenshots as **negative whitespace mid-page** at narrow widths. Density needs another pass: not because it's too sparse, but because the _vertical rhythm_ keeps stuttering between gap-2 / gap-3 / gap-6 inside the same column.

Naming has gentle drift. The sidebar says **Deadlines**, the page header says **Deadlines** (wrapped in a `ConceptLabel` that pops up an info popover), the API contract says **obligations**, the file is `obligations.tsx`, the search filter URL uses `?status=`, the detail drawer URL uses `?drawer=obligation&id=…`, and the dashboard subhead reads "Actions this week" (not deadlines). A CPA new to the app will hit _deadlines_, _obligations_, and _actions_ in three clicks for one concept. The wizard-copy audit already covered the front door — this one calls out the **navigation layer**: the leftnav label, the page title, the URL noun, and the user-facing tile copy should all settle on **Deadlines**.

Cross-surface inconsistencies layered on top:

- Page-header padding: `/clients` and `/deadlines` use `p-3 md:p-5`; `/` (dashboard) uses `p-4 md:p-6`; `/opportunities` uses `p-3 md:p-5`; `/rules/pulse` uses `p-3 md:p-4` (and switches between `max-w-page-wide` and `max-w-[1440px]` when the side panel opens). Five values, no rule.
- Outer page gap: `/clients` and `/deadlines` use `gap-4`; `/` uses `gap-6`; `/opportunities` uses `gap-4`; `/rules/pulse` uses `gap-4`. Three values.
- "Open deadlines" count chip: rendered as `bg-state-base-hover` rounded-full on `/clients` h1; as `text-text-tertiary tabular-nums` plain text on the dashboard `Actions this week` h2; as a separate `rounded-md border` `StatTile` body on `/rules/library`.

Finally, the **client detail page's 4-tab IA** (`Work / Client info / Discover / Activity`) is the right call — but the Work tab now duplicates summary information that lives 100 px above it (the `ClientSummaryStrip` already shows "Next due", "At risk", "Open filing"; the Filing-plan section below adds an h2 + subtitle that says "N deadlines across N tax years"). Two summary headers, two depths of count. This is the single biggest readability liability on the detail page after the tile zoo.

## 3 · Per-surface evaluation

### 3.1 — /clients (list) · `ClientFactsWorkspace` + `routes/clients.tsx`

#### Macro evaluation

The page does one job — let a CPA scan the practice directory, narrow it, and drill into a client — and the structural backbone supports that goal well. The table sits flat against the page background (no Card chrome, no inner border), filter toolbar is one row above it, action strip (banner + 3 tiles) sits above that, page header tops the whole stack. Four distinct chrome elements compete for the eye on first paint: PageHeader, NeedsFacts banner, three stat tiles, ToolbarFilters row. That's two-many for a list page — the filter toolbar and the stat tiles are doing **filter** work (just different filter dimensions: counts vs facets), so the visual separation between them is the wrong communication. Information density is good once you reach the table, but the path from page-load to first row scan crosses too many distinct surfaces. Brand consistency reads as part of the same product as `/deadlines` (same hairline table, same column-sort buttons, same monogram avatars).

#### Detail findings

**L1. The action strip + filter toolbar read as two separate "filter areas" but are functionally the same — (`ClientFactsWorkspace.tsx:1175-1216`)**

- _What:_ `ClientsActionStrip` renders 3 stat tiles ("At risk", "Waiting on client", "Pulse hits") which act as URL-state filter toggles. Below it, `ClientsFilterToolbar` renders 4 facet filters ("Client", "States", "Entity", "Owner"). Both are filters. Both can be active simultaneously. The visual separation (8-px gap, different shapes, different active treatment) makes them feel like unrelated controls.
- _Why it hurts UX:_ CPA opens page, has to learn two separate filter languages. The "active filter" state on a tile (border-accent + bg-accent-hover, `:1647`) doesn't visually match the active state on a toolbar trigger (count chip). And the Reset button only resets toolbar filters — atRisk + waiting remain active. The user has to know there are two reset surfaces.
- _Fix:_ Either (a) move the 3 tone-coded tiles into the toolbar row as iconified chips (matches `/deadlines`' action-chip strip at `obligations.tsx:2301-2334`), or (b) keep the tiles separate but unify Reset to clear _all_ table filters including atRisk/waiting state.
- **Sev:** P1

**L2. The 220-px primary-state pill ("CA · California") is heavier than every other cell — (`ClientFactsWorkspace.tsx:800-820`)**

- _What:_ `States` column renders the primary state as a full `rounded-full border` pill with the StateBadge SVG + 2-letter code + full state name. Other-state badges in the same cell are bare SVG motifs. Cell width is 220 px (was widened from 160 px specifically to fit "California").
- _Why it hurts UX:_ The pill is the heaviest visual treatment in the row — heavier than the client name (which is the row's identity), heavier than the next-due relative label, heavier than the owner avatar. At a glance the eye lands on the state pill first, not the client name. The pill's job is to communicate jurisdiction; that doesn't need the full state name on a list view where the user already filtered to one state or where the 2-letter code suffices (CPAs read state codes fluently).
- _Fix:_ Drop the full state name from the pill in the list view. Keep the StateBadge SVG + 2-letter code in a smaller pill (h-5, padding-x-1.5). The full name belongs on the detail page (where there's space) and on hover (via the existing `title` attribute). Reclaim the 100 px for the name column or for an aspirational "Last contact" date column.
- **Sev:** P1

**L3. "Next due" cell is a 2-line composite — relative + ISO + status pill — and the ISO date is the second weakest line — (`ClientFactsWorkspace.tsx:897-918`)**

- _What:_ Cell renders `<NextDueRelativeLabel>` ("12d late" / "today" / "in 3d") on line 1, then ISO date "2026-05-23" + status pill on line 2.
- _Why it hurts UX:_ The ISO date is the column's most precise signal but reads as caption-tier text below the relative phrase. A CPA who knows a deadline is "12d late" wants to immediately match it to a calendar — having the date subordinate to the phrase forces them to read both lines. The mixed font weights (relative is medium tabular, ISO is monospace tabular) split the eye further.
- _Fix:_ Reverse the visual hierarchy: ISO date primary (`text-sm tabular-nums text-text-primary`), relative phrase secondary (`text-xs text-text-warning` / tertiary). Status pill on line 2. This matches how the obligations queue treats the same data (which the filing-plan section borrows). Same data, same shape, every surface.
- **Sev:** P2

**L4. Pagination footer only renders when count > 25, hiding the row total — (`ClientFactsWorkspace.tsx:1305-1340`)**

- _What:_ The "N clients · pages 1-of-N" footer only appears when there are >1 pages of clients. For a firm with ≤25 clients, no footer renders.
- _Why it hurts UX:_ The header has a count chip ("9 Clients") but the bottom of the table is silent. After filtering (which can leave 0-1 rows visible), the user has no answer to "how many actually matched?" without scrolling back up. The table's empty-row state at `:1671-1685` helps when count is 0, but partial-match states (e.g. 3 of 9 shown) get no feedback.
- _Fix:_ Render the count + Reset link in a small footer strip on every page, even when there's only one page. `Showing 3 of 9 clients · Reset filters`. Matches `/deadlines`' "N obligation(s)" footer treatment at `obligations.tsx:2814-2817`.
- **Sev:** P2

**L5. The peek-icon hover affordance is invisible until hover — (`ClientFactsWorkspace.tsx:746-756`)**

- _What:_ Each row has an `EyeIcon` button at the right edge of the name cell that opens `ClientPeekHoverCard` (the read-only side drawer). It's `opacity-0 group-hover:opacity-100`.
- _Why it hurts UX:_ Discoverability is zero — a CPA using keyboard never knows it exists, and a CPA on a touch device can't trigger hover. The keyboard shortcut (`⌘-click` on the row, per `:1261-1266`) is even less discoverable. The peek feature is genuinely useful, but it's invisible.
- _Fix:_ Show the icon at low opacity (`opacity-30 group-hover:opacity-100`) as a persistent affordance. Or fold the row's keyboard target into a "Press Space to peek, Enter to open" hint in the keyboard-shortcuts dialog. Or add the icon to the column header tooltip ("Hover any row → click eye to peek").
- **Sev:** P2

**L6. Banner verb is "Fix now" with `variant="destructive"` — too aggressive for a setup step — (`ClientFactsWorkspace.tsx:1561-1563`)**

- _What:_ The `NeedsFacts` banner has a primary CTA "Fix now" rendered with `variant="destructive"` (red). The banner itself uses `variant="warning"` (amber).
- _Why it hurts UX:_ Two different alarm levels in one component (amber surrounding + red button) signals an ambiguous severity. The action isn't destructive — it's setup completion. The red button on amber is also a known a11y trip: AA contrast between destructive button text and the warning bg is borderline. The code comment explicitly justifies the choice as "missing facts means the rule library is silently skipping these clients, which IS a destructive outcome" — but that's reasoning about the _consequence_, not about the action the user is taking.
- _Fix:_ Switch the button to `variant="default"` (primary accent). Banner already carries the urgency via the amber surround + AlertTriangle icon. If you want extra weight, increase the button size from `sm` to `default`. The pull-the-pin warning is already loud enough.
- **Sev:** P2

**L7. ClientsCreateSplitButton + Archive button + page-count chip = three different "right edge" treatments — (`routes/clients.tsx:303-345`)**

- _What:_ PageHeader actions cluster is: `<Button variant="outline" Archive>` + `<ClientsCreateSplitButton>`. Title slot has a count chip. Three different shapes (outline button, split button, rounded-full chip) compete for the right edge of the top row.
- _Why it hurts UX:_ Eye load on the page is heavier than it needs to be. The Archive button's purpose ("imports archive — review past CSV migrations") is hidden under a title attr; first-time users won't decode it. The count chip is `rounded-full bg-state-base-hover` which is the only rounded-full element on the page (other badges use `rounded-md`).
- _Fix:_ Move "Imports archive" into an overflow menu (`MoreHorizontalIcon`) next to "+ Add client". Drop the count chip from the title and let the footer/toolbar own the count signal. Now the right edge has one primary button (Add) and one overflow — clean GitHub rhythm.
- **Sev:** P2

**L8. Stat-tile sublabels were dropped to match the rule library — but the rule library's tile rhythm uses border + padding-x-3 + smaller value (`text-xl`), while the /clients tile is also `text-xl` but with no sublabel and centered focus. They are visually nearly identical but built independently — (`ClientFactsWorkspace.tsx:1606-1659` vs `rules.library.tsx:1266-1310`)**

- _What:_ `ClientsStatTile` is a button or div with caption-tier uppercase label + tabular numeral value. `StatTile` in rules.library is structurally identical but with optional `sublabel` and `href` link variant. Code comments at `:1576` explicitly say "to match the rule library StatTile exactly" — but the rules-library StatTile is not exported from a shared module.
- _Why it hurts UX:_ Future style drift is locked in. When someone updates one tile, the other doesn't follow. This is the most obvious "extract a shared primitive" call in the audit.
- _Fix:_ Extract `apps/app/src/components/patterns/stat-tile.tsx` with the shape (label / value / tone / optional sublabel / optional href or onClick / optional `active` pressed state). Migrate `/clients`, `/clients/[id]`, `/opportunities`, `/rules/library`. ~2 hours; eliminates the entire family of inconsistencies in audit item #1.
- **Sev:** P0 (cross-surface)
- **Status (2026-05-26):** **Shipped** — see top-10 #1 row above. The audit's "5 implementations" became 2 by ship time (PR #25 + the seventy-second pass retired the `/clients` and `/rules/library` tiles ahead of the extract). `optional sublabel` and `active pressed state` were dropped from the primitive's API because the surviving consumers don't use them — kept simpler; restore-from-git when a future consumer needs them. The `/clients/[id]` `ClientSummaryStrip.TileShell` deliberately did NOT migrate (Figma-replica anchor tile with different design intent; documented inline).

### 3.2 — /clients/[clientId] (detail) · `ClientDetailWorkspace`

#### Macro evaluation

The detail page is doing two jobs honestly: surface client-level signals ("anything wrong right now?") and provide the daily work surface (Filing plan). The 4-tab IA (`Work / Client info / Discover / Activity`) does that — Yuqi's earlier IA decision was correct. The visual problem is that the page tries to do everything **above** the tabs as well: PageHeader (with eyebrow back-link, title with switcher + 5 chips, subtitle, actions cluster) → ContactMetaRow → ActiveAlertsSection → SummaryStrip → Tabs. That's 5 distinct horizontal blocks before the user reaches the first piece of tabbed content, and each one uses a different visual treatment (h1 + chips, link icons, framed alerts list, off-white tiles, tab underline). When the obligation drawer opens to the right, the entire stack compresses further into ~60% width. The dominant action — "open today's overdue filing" — is buried below the fold on most viewports. Information density is _too dense_ above the fold and _too sparse_ inside Activity / Discover tabs. Brand reads consistent with the rest of the app, but the page is the densest in the whole product.

#### Detail findings

**D1. H1 chip cluster is 4-7 chips inline with the title; wraps to a second row below md — (`ClientFactsWorkspace.tsx:2005-2068`)**

- _What:_ Inside the H1: `<ClientTitleSwitcher>` (client name dropdown), `<Badge variant="outline">` (entity type), `<ClientOwnerHeaderPill>` (avatar + name + chevron, opens dropdown), `<ClientFilingStateChips>` (1-3 mono state codes + overflow), conditional `<Badge variant="destructive">` (Add filing state). At default desktop width that's 5 distinct interactive elements + the title itself, all on one row.
- _Why it hurts UX:_ The h1 row has become an "everything bar" — title + identity + ownership + setup state + setup CTA. Tab order is unclear (5 focusable items in one row), the visual midline keeps shifting because the chips have different vertical centers (small text inside a button vs medium semibold name vs mono uppercase code). A user scanning the page can't find the title quickly because it's surrounded by 4 other things at the same scale.
- _Fix:_ Split into two rows by responsibility: row 1 = title + switcher only (h1 owns its own width), row 2 = identity + owner pill + filing-state chips as a 12-px-text "metadata strip" with `text-text-secondary`. Add-state CTA moves to the actions cluster on the right edge. Now h1 reads as a single title; chips read as supporting metadata; the missing-state CTA reads as a real action.
- **Sev:** P0
- **Status (2026-05-26):** **Shipped via PR #25** (`design(clients): directory pivot — phases 1-3`, commit `374dcd23`). The fix landed at the page-body tier rather than as a PageHeader slot — chips moved to `ClientContactMetaRow` (rendered between the PageHeader and the SummaryStrip), and the readiness CTA went from `destructive` to `warning` variant. The h1 cluster now carries just `ClientTitleSwitcher` + (conditional) readiness chip. A canonical `PageHeader.metaRow` slot was added in `docs/dev-log/2026-05-26-client-detail-filing-plan-dedup-and-metarow.md` for future routes with the same pressure (R3, members-page); `/clients/[id]` stays on its body-level ContactMetaRow since the shape is already working.

**D2. PageHeader actions cluster has 3 buttons, all icon-only, all unlabeled — (`ClientFactsWorkspace.tsx:2097-2116`)**

- _What:_ `<ClientHeaderOverflowMenu>` (`···`), `<Button icon-sm Archive>` (file cabinet), `<CreateObligationDialog>` (Add deadline, has label inside).
- _Why it hurts UX:_ The overflow menu currently has exactly ONE item ("View audit log" if permitted) per the comments at `:3303-3318`. That's a 1-item overflow menu sitting next to an icon-only Archive button — the user has to hover twice to learn what the page can do. The "+ Add deadline" button is labeled but its position next to two icon-only buttons makes the row visually unbalanced (left two = quiet glyphs, right one = labeled CTA).
- _Fix:_ Inline the overflow's single audit-log item as a labeled outline button OR remove the overflow menu when it has ≤1 item. Add a tooltip text label to Archive ("Archive client"). Or label-up the Archive button as `<Button variant="outline" size="sm"><ArchiveIcon /> Archive</Button>` — matches the /clients list page's labeled Archive treatment (#L7).
- **Sev:** P1

**D3. ActiveAlertsSection has its own framed surface — bordered, padded, with a tone-coded header — that disrupts the page's flat body — (`ClientFactsWorkspace.tsx:3214-3249`)**

- _What:_ The section renders inside a `border + bg-background-default + p-4` frame with a tone-tinted header bar (`bg-components-badge-bg-warning-soft/40 px-4 py-2.5`). The page's other content blocks (`TabSection`, Tabs body) are deliberately frameless. Reads like a v1 design pasted into a v2 page.
- _Why it hurts UX:_ The frame is the heaviest content surface above the fold — heavier than the page header. When zero pulse alerts are active (the common case), the frame disappears entirely; when alerts ARE active, the framed block grabs visual priority that probably belongs to the Filing plan. Worse: the header includes `<MegaphoneIcon className="size-3.5">` next to "Active alerts for this client" inside an uppercase eyebrow — emoji-grade typographic density.
- _Fix:_ Adopt the same flat treatment the rest of the page uses: a `TabSection`-style header (`h2 text-base font-semibold text-text-primary` + subtitle), no frame, no tinted header bar. The alert cards themselves carry the urgency tone — the section wrapper shouldn't double down. Optionally: when ≥1 alert is active, surface a small destructive `BadgeStatusDot` next to the section title; the dot communicates state without re-framing the whole block.
- **Sev:** P1

**D4. SummaryStrip "Open filing" tile + Filing-plan h2 subtitle restate the same count — (`ClientSummaryStrip.tsx:318-325` + `ClientFactsWorkspace.tsx:2710-2716`)**

- _What:_ `ClientSummaryStrip` renders a tile labeled "Open filing" with the count of non-terminal obligations + sublabel ("N forms in motion"). 200 px below, the Filing plan section's subtitle reads "N deadlines across N tax years". Same denominator, different phrasing, two different reading positions.
- _Why it hurts UX:_ The user reads the same fact twice with different copy. Worse, the tile's count is "open" (non-terminal) and the subtitle is "all" (including terminal years) — but the visual treatment doesn't communicate that distinction. A user looking at the page sees "5 open filing" in one place and "12 deadlines across 3 tax years" in another and doesn't know which to trust.
- _Fix:_ Drop the "Open filing" tile and make the third summary slot something the page doesn't already say (e.g. "Last filing closed" or "Documents needed"). OR keep the tile and remove the count from the Filing plan subtitle. Don't render the same number twice 200 px apart with different denominators.
- **Sev:** P0
- **Status (2026-05-26):** **Shipped via the second path.** The `ClientSummaryStrip` "Open filing" tile is now the single owner of the count (already half-canonical per the 2026-05-24 distill pass that dropped "N open filings" from the workPlan summary). Filing-plan h2 subtitle now reads `Grouped by tax year, newest first` (≥2 years) or `Latest first` (1 year) — structural fact only, no count. The tile stays because it's a click-target (opens `/deadlines?client=<id>`); dropping it would lose interaction value with no obvious replacement signal. See `docs/dev-log/2026-05-26-client-detail-filing-plan-dedup-and-metarow.md`.

**D5. Filing-plan year cards use `rounded-xl bg-background-soft` inside a flat `TabSection` inside a flat page — three nested visual contexts — (`ClientFactsWorkspace.tsx:2968`)**

- _What:_ Each year section is wrapped in `rounded-xl border border-divider-subtle bg-background-soft`. Inside the wrapper is a year header bar + column header bar + rows. The `TabSection` itself is frameless (h2 + subtitle only). The page body is frameless. So the page is frameless → tab body is frameless → section header is frameless → year card has 12-px radius + grey fill.
- _Why it hurts UX:_ The year card frame is the only chunky tile on the page. It reads as a v1 design surface inside a v2 page. The column-header bar repeats per year (`:3019-3072`), so when a client has 3+ tax years the user scrolls past 3 column-header bars to get to ancient history. The code comment at `:2719-2728` acknowledges the trade-off but accepts it for the sake of self-contained year cards.

- _Why it hurts UX:_ The collapsed dismissals card stands out as the only framed surface on the page. When collapsed, it's a 60-px-tall framed strip that competes with the actual content above.
- _Fix:_ Drop the Card wrapper. Render as a flat row with the chevron + title + count + helper text. Expanded state shows the dismissed rows beneath inline.
- **Sev:** P3

### 3.8 — Sidebar · `app-shell-nav.tsx`

#### Macro evaluation

The sidebar has the simplest job and does it well. Firm-switcher header → primary destinations (Today / Alerts / Deadlines) → Rule library → Clients group (Clients / Opportunities) → footer (Alerts archive / Audit log / Settings, dimmed). Icons are well-chosen (Calendar for Today, Megaphone for Alerts, SquareChartGantt for Deadlines, BookOpen for Rule library — the icon-set commentary at `:561-582` explains the picks). Badge tones — urgent (pulse) and inventory (deadlines / clients counts) — distinguish "look at this" from "reference fact" clearly.

Macro problems are minor: the group labels (`Operations`, `Rule`, `Clients`) sit between the primary items but are inconsistent with v2's actual structure (where `Today / Alerts / Deadlines` are standalone primary items without a group label, then `Rule` is its own group label, then `Clients` is its own group label). The result is one labeled group (Rule), one labeled group (Clients), and a footer that has no label but is dimmed — and the navV2 branch deliberately drops the `Operations` label but keeps the primary items above (`:574-592`). Yet the legacy branch keeps `Operations`. So depending on the feature flag, "Today / Alerts / Deadlines" lives inside an `Operations` group or as a label-less standalone trio.

#### Detail findings

**S1. Group-label inconsistency between navV2 and legacy — (`app-shell-nav.tsx:574-642` vs `:646-703`)**

- _What:_ navV2 puts primary trio above a `Rule` group (one item) and a `Clients` group (Clients + Opportunities). Legacy puts everything inside `Operations`, `Clients`, `Rules` groups.
- _Why it hurts UX:_ The feature flag flip changes the visible structure of the sidebar in a perceptible way. Users testing one variant will report a different nav shape than users testing the other.
- _Fix:_ Pick one shape (v2 is cleaner — primary items don't need a group label) and remove the other branch. The feature flag has been at the gating layer long enough; promote.
- **Sev:** P2

**S2. "Alerts archive" sits in the dimmed footer next to Audit log — discoverability cost — (`app-shell-nav.tsx:626-641`)**

- _What:_ `/rules/pulse/history` is in the footer at 55 %-opacity (`mt-auto opacity-55`). The footer also holds Audit log and Settings.
- _Why it hurts UX:_ The Alerts archive is the closed-state for /rules/pulse — i.e., "where did the alert I dismissed last week go?" A user searching for a dismissed alert won't find the dimmed footer link. Audit log + Settings being dimmed is right (rarely-used surfaces); Alerts archive belongs _adjacent to_ the live Alerts entry above, not at the bottom of the sidebar.
- _Fix:_ Move Alerts archive directly under Alerts as a sub-item (or as a second item in a small label-less group), or surface it as a "View archive" link on the Alerts page itself rather than in the sidebar.
- **Sev:** P2

**S3. Firm switcher trigger is 56 px tall — heavier than every other sidebar row — (`app-shell-nav.tsx:202-235`)**

- _What:_ The `<button h-14 ...>` is the firm switcher trigger. Sidebar nav rows are ~40 px tall.
- _Why it hurts UX:_ The trigger is a 1.4x taller surface at the top of the sidebar. On a 13" laptop it claims ~7% of the sidebar height. The 32-px monogram avatar (rationale at `:214-220`) is correct for brand identity but the surrounding padding doesn't need to be h-14.
- _Fix:_ Shrink to `h-12` (48 px) — still tall enough for the 32 px avatar with breathing room, but tighter against the rest of the rail.
- **Sev:** P3

### 4.2 — PageHeader adoption (6 of 8 surfaces use it, 2 opt out)

Both opt-outs lose the canonical eyebrow tracking + `text-2xl leading-7` h1. Both gain a custom h1 (dashboard's 20 px, Pulse's PulsingDot-prefixed 24 px). Inconsistent header scales = inconsistent perceived page importance. **Fix:** route both through `<PageHeader>`; both already accept ReactNode `title` so the customizations slot in.
