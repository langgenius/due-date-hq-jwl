# 我们是怎么工作的

> 工作流里堆的工具越多,代价越大。

---

## 方法

### 目前的设计工作流

用的工具:Claude Code、Pencil。

1. 铅笔画线框图。
2. AI 出一个能跑的初稿——真实组件、真实数据、真实约束。
3. 调研:产品模型、CPA 领域、竞品,以及后端到底能支持什么。
4. 接到 Pencil dev(或其他可视化面板)精修:把 key screens 一起展示,交叉对比、修改。
5. 整体翻修之后,回 Claude Code 改代码。
6. 整体过一遍 design audit,确认组件库用对了。

重复 3–6。

### 设计起点

1. 有个已经差不多能运行的 demo。
2. 没有定稿的信息架构(IA),边做边定。
3. 没有 Figma。
4. 一开始不确定用户旅程——现在有大概的旅程,但需要验证。
5. 不确定每个页面有什么。
6. 直到昨天才有设计系统。
7. 没有用户测试。

### 现在有了什么

1. 一套代码化的设计系统:token、组件、按钮、模式。`DESIGN.md` 是单一参考,`/preview` 是组件库。
2. 主要页面都成型:today、deadlines、clients、alerts、rules library。
3. 跨页面一致性:同一个数据各页面共用一处,不让数字打架。
4. 状态体系压缩成 6 个状态(迁移还在收尾)。
5. V1 品牌落地。
6. 中英双语(i18n)。
7. 设计 / 产品决定都有记录:约 70 篇设计文档、约 1240 篇 dev-log。
8. 约 21 万行 TypeScript 源码(不含测试和构建产物)。

### 一半在去噪和梳理

- 把散落、写法各异的手写样式抽成 primitive,所有页面统一。
- 文案先两个角度各审一遍再改:IA 一遍、UX 写作一遍。
- 每隔一阵把偏移的视觉扫一遍。
- 每次大改在 Pencil 留 node ID,反着指回实现。

### 一半在增加设计细节

- 截图 → 挑毛病 → 改 → 再截。
- 改链路:① 状态跨页流转;② 每个东西在哪打开。

### 知道什么时候停

- 一旦觉得 OK 就往前做下一个任务,之后再来改。
- 锁定决定、记下理由、不再翻来覆去改。
- Drift audit 胜过无止境打磨——最终都要收敛。
- 需要更多时间吗?其实永远能再改下去——所以得有个明确的时间节点逼自己停(很多时候就是 demo 到了)。

## 设计上的决定

一些具体、有取舍的设计判断:

- 灰色 =「没事」,不是绿色;绿色只留给已申报 / 已完成。
- 紧急只用尺寸和颜色,绝不「红 + 加粗」。
- status 是观察出来的,不是下拉框里选的。
- Filed ≠ Done:payment 逾期是独立信号,不被「已申报」盖掉。
- 风险先讲美元、再讲天数。
- AI 输出没有来源就不渲染——宁可空,不可编。
- 一个模式只有一个 primitive,不手搓;一页最多一张卡。
- 整行可点,没有「Review」按钮——扫一眼就能点开。
- 选中态安静、筛选开启态响亮——两种不同的高亮。
- 数字一律对齐,圆角封顶 8。

## 工作真实的样子

### 设计并不完美

- 到了「还行」就很难再往前推一步——容易停在及格线。
- 就算有稿,要一比一、又聪明地实现、还不破坏设计系统,本身就难——做出来的和想要的总有差距。
- 有时候根本不知道要什么,就让 AI 随便做了——做出来什么就是什么,其实并不好。

### 什么时候需要设计系统?

在第一轮(round 1)结束的时候——就像现在,昨天才有。一开始立不起来,因为还不知道产品长什么样。

### 设计决策一直在漂

- 当你没有什么可以依据的时候,设计从来不是线性的——决策上会有很多漂移和来回。
- 判断的循环:deadline 详情页的 tab 数从 6 → 4 → 3 → 4 → 3 → 4 才锁定。视觉上也在漂(这里一个「显得太细」的 `text-2xl`,那里一个「和 h1 抢」的 `text-lg`)。
- 而且如果组件或 token 没搭好,改一个地方来保证一致就会很痛。很多东西是藏着的,你也忘了。

### 取舍

调整过程中会有一些代价更大、不适宜现在去做(修正)的内容,一定要 log,并且记得回顾。

### 大部分工作是看不见的

大部分时间都花在看不见的地方:drift audit、改名、第三遍一致性梳理。

## 版本管理——直接提 main,但有纪律

我会用 git,但从没真正在一个团队结构下好好工作过(虽然组员 = 1 个)。

- 开多个 session、多个 agent,以为各自独立做一部分——其实在同一个文件夹里共享文件系统,会重叠、互相覆盖。
- 说「有纪律」,但有时候更像在疯狂 commit——真正的团队协作长什么样,我还没经历过。

- 质量全压在一个人眼睛上,没有第二个 reviewer。
