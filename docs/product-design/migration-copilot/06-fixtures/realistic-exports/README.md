# Realistic CPA Export Fixtures

Generated by `node scripts/generate-migration-realistic-fixtures.mjs`
(guarded by `node scripts/validate-migration-realistic-fixtures.mjs`).

These fixtures simulate export files a CPA practice would upload into `/migration/new`.
They use the same 24 synthetic clients across all vendors and reshape file names, formats,
headers, and column order to match each source product's **real** export — verified against
official vendor documentation. All PII is fake: client names include `(TEST)`, emails use
`example.com`, phones use `555-01xx`, and EIN-like identifiers use the `99-*` test range.

The header sets, ID formats, and value domains below are modeled on what each product actually
exports (see Source Research Links). `detectSource()` in
`apps/app/src/features/migration/intake-files.ts` keys on these realistic signatures.

| `ultratax-client-listing-report.dif` | Minimal UltraTax DIF stub (rejected by extension)    | Rejected with `ultratax_dif` guidance: re-save the Client Listing Report as CSV/XLSX         |
| `file-in-time-backup.fbk`            | File In Time backup marker                           | Rejected with `file_in_time_backup` guidance: export Task View or Client Information instead |

> `.dif` and `.fbk` are intentionally minimal byte stubs — they exist only to exercise the
> "reject by extension + show export guidance" path, not to be parseable exports.

## Data Shape

- 24 rows per accepted fixture; every row is clean and valid (no missing states, typo states,
  or non-existent return types).
- Mixed entities: `LLC`, `Partnership`, `C-Corp`, `S-Corp`, `Individual` (incl. sole proprietors
  filing Schedule C as a `1040`), `Trust`, `Nonprofit`.
- Mixed return types: `1040`, `1065`, `1120`, `1120S`, `1041`, `990`.
- Mixed states: `CA`, `NY`, `TX`, `FL`, `WA`, `IL`, `CO`.
- Realistic per-product fields and formats:
  - Return types display as bare form numbers (`1065`), not `Form 1065`.
  - Staff/preparers are named people (`J. Doe`), not login codes.
  - TaxDome account IDs are name-initials + a number (`MH1`); custom CRM fields appear as their
    own columns (no `Custom field -` prefix); account status uses the activation vocabulary.
  - Karbon keys are opaque alphanumerics; the export has no tax-id, tags, or notes columns
    (those require a Karbon Support request).
  - QuickBooks Desktop IIF opens with an `!HDR` section, uses the full `!CUST` field set, and
    REFNUM is QuickBooks' internal sequential integer.
  - CCH Axcess uses `Client Name` / `SSN/FEIN` / `Client Type` / `Sub-ID` and `Active`/`Inactive`
    status; CCH ProSystem fx emits the Portal batch client-linking schema (mandatory `Client GUID`,
    `NameLine1`/`NameLine2`, `SortName`, `ClientType`, `FYE`).
  - Lacerte uses real selectable field labels: `Client Full name`, `Federal ID Number`,
    `Type of Entity`, and spelled-out telephone labels.
  - UltraTax models the **Client Contact** listing report (the only one carrying email/phone).
  - ProConnect populates `Taxpayer name` for individual returns and `Business name` for entities.
- File In Time due dates are extended deadlines per return type: 1065 / 1120-S = 09/15,
  1040 / 1120 = 10/15, 1041 trust = 09/30, 990 = 11/16 (weekend-adjusted).

### Deliberate simplifications

A few fixtures trade strict fidelity for app/test coverage; these are intentional:

- **TaxDome bundling.** Real TaxDome emits a _separate_ ZIP per data type. We keep one ZIP
  containing `accounts.csv` + `contacts.csv` because the app's cross-CSV merge
  (`buildTaxDomeBundleCandidate`) operates within a single ZIP.
- **Karbon `.xlsx`.** Karbon's self-serve contact export is CSV; we keep `.xlsx` to retain
  xlsx-parser coverage (Karbon also surfaces spreadsheet downloads).
- **QuickBooks Online title rows.** A raw QBO report export has company/title rows above the
  column headings (users delete them); we start at the heading row, which the app parses as row 1.
- **Lacerte single module.** Lacerte exports per module (Individual / Partnership / Corporate);
  this fixture concatenates modules into one `EXPORT.CSV` using a shared `Federal ID Number` label.
